## 1. 文件目的與範圍

### 1.1 目的

本文件為「客戶 360 度戰情室（CUS360）」系統之系統需求規格書（System Requirements Specification, SRS），供後續正式開發人員依此進行系統設計、功能實作與測試驗收。

### 1.2 範圍

CUS360 是一套銀行內部使用的客戶關係管理（CRM）戰情室平台，整合客戶基本資料、財務資產、產品持有、交易行為、標籤分析與 AI 建議等功能，提供理財專員與主管在單一介面掌握客戶全貌，並支援個資遮罩、角色分權與資料匯出。

### 1.3 適用對象

- 系統開發人員（前端 / 後端）
- 測試工程師
- 系統架構師
- 資訊安全審查人員

## 2. 系統概述

### 2.1 系統定位

CUS360 為銀行行員使用的內部客戶智能平台，目標在整合各系統客戶資訊，呈現 360 度客戶視圖，協助前線業務人員進行精準服務與業務推薦。

## 3. 使用者角色與存取控制

### 3.1 角色定義

系統支援兩種使用者角色，於登入時選定，影響可使用之模組與功能。

| 角色代號 | 角色名稱 | 說明 |
|----------|----------|------|
| `manager` | 主管（經理） | 完整存取所有模組，包含儀表板與全客戶資料 |
| `specialist` | 理財專員 | 存取客戶查詢、篩選名單、客戶詳情；不含儀表板 |

### 3.2 模組存取矩陣

| 功能模組 | 主管 | 理財專員 |
|----------|:----:|:--------:|
| 客戶查詢 | ✅ | ✅ |
| 篩選名單 | ✅ | ✅ |
| 儀表板 | ✅ | ❌ |
| 客戶詳情（所有分頁） | ✅ | ✅ |
| 資料匯出 | ✅ | ✅ |
| 智能小助手 | ✅ | ✅ |
| 個資遮罩切換 | ✅ | ✅ |

> **備註**：目前版本個資遮罩為使用者主動切換（非強制角色鎖定）。若未來合規需求升級，理財專員應強制遮罩敏感欄位。

### 3.3 登入認證規格

**需求 AUTH-001**：系統啟動後呈現登入畫面，使用者輸入帳號（loginUser）與密碼（loginPass）後方可進入主功能。

**需求 AUTH-002**：登入成功後需記錄 `currentRole`（`manager` 或 `specialist`），並依此控制模組顯示與功能存取。

**需求 AUTH-003**：登入過程中顯示載入動畫（建議時延 800ms），提供使用者視覺回饋。

**需求 AUTH-004**：登入失敗時顯示錯誤提示訊息，不洩漏具體失敗原因（如「帳號或密碼不正確」）。

**需求 AUTH-005**：登出功能需清除所有 session 狀態，返回登入畫面。

### 4.2 客戶查詢模組（Search）

#### 4.2.1 功能概述

系統提供單一客戶查詢入口，支援多種查詢條件，結果導向客戶詳情頁面。

#### 4.2.2 查詢類型與格式驗證

**需求 SEARCH-001**：系統支援下列四種查詢類型，使用者可手動選擇或系統自動偵測輸入格式：

| 查詢類型 | 欄位 | 格式規則 | 匹配邏輯 | 輸入範例 |
|----------|------|----------|----------|----------|
| 身分證字號 | `idCard` | 1 英文字母 + 9 位數字（共 10 碼） | 完全匹配 | `A123456789` |
| 信用卡號 | `creditCard` | 16 位數字，可含分隔符（-）| 正規化後完全匹配 | `4559-1234-5678-9012` |
| 存款帳號 | `accountNumber` | 格式 `XXX-XXXXXXX-X` | 正規化後完全匹配 | `004-1234567-1` |
| 姓名 | `name` | 中文或英文字串 | 模糊匹配（包含即可） | `王小明` |

**需求 SEARCH-002**：查詢執行時顯示載入狀態（Spinner），建議超時時間 800ms，超時後顯示查無結果或錯誤訊息。

**需求 SEARCH-003**：查詢結果若唯一，自動開啟客戶詳情頁；若多筆，顯示結果列表供使用者選擇。

**需求 SEARCH-004**：系統應記錄使用者最近 5 筆查詢歷史（`queryHistory`），並提供一鍵重複查詢。

### 4.3 篩選名單模組（Filter）

#### 4.3.1 功能概述

提供多條件複合篩選，支援簡易篩選與進階標籤布林邏輯篩選，篩選結果可匯出。

#### 4.3.2 篩選條件

**需求 FILTER-001**：系統支援下列基本篩選條件（單選或多選）：

| 條件名稱 | 欄位 | 可選值 |
|----------|------|--------|
| VIP 等級 | `vipLevel` | VVVIP、VVIP、VIP、Normal |
| 風險等級 | `riskLevel` | low、medium、high |
| 帳戶狀態 | `accountStatus` | active、失效戶 |
| 行銷同意 | `marketingOptIn` | 是、否 |
| 所在城市 | `city` | 多選，台灣主要城市清單 |

**需求 FILTER-002**：系統支援進階標籤篩選，可建立多個條件群組，群組內條件以 AND / OR / NOT 邏輯組合，群組間以 AND / OR 連接：

```
篩選邏輯範例：
群組 1：(標籤A AND 標籤B) OR (NOT 標籤C)
群組 2：(標籤D AND 標籤E)
最終結果：群組1 AND 群組2
```

**需求 FILTER-003**：標籤篩選支援下列五種標籤分類：

| 分類代號 | 分類名稱 |
|----------|----------|
| `意圖標籤` | 客戶行為意圖（如：出國旅遊意圖、房貸需求） |
| `客戶屬性` | 客戶靜態屬性（如：高資產戶、新開戶） |
| `交易互動` | 交易行為特徵（如：高頻交易客戶） |
| `風險警示` | 信用風險訊號（如：逾期記錄、信用狀態觀察） |
| `消費習慣` | 消費偏好分類（如：旅遊高消費、醫療費用偏高） |

**需求 FILTER-004**：篩選執行後顯示結果筆數與符合條件摘要。

### 4.4 儀表板模組（Dashboard）

#### 4.4.1 存取限制

**需求 DASH-001**：儀表板模組僅限 `manager` 角色存取；`specialist` 角色不顯示此模組入口。

#### 4.4.2 KPI 指標

**需求 DASH-002**：儀表板應顯示下列全行客群統計指標：

| 指標名稱 | 計算說明 |
|----------|----------|
| 客戶總數 | 系統管理之全部客戶筆數 |
| 有效戶數 | `accountStatus = active` 之客戶數 |
| VIP 等級分佈 | 各等級（VVVIP / VVIP / VIP / Normal）佔比 |
| 產品貢獻佔比 | 信用卡 / 放款 / 存款 / 投資 四項貢獻度分佈 |
| 信用指標 | DPD30 逾期率、不良貸款率（NPL）、平均信用卡使用率 |
| 收益趨勢 | 月度收益趨勢折線圖（最近 12 個月） |

#### 4.4.3 產品佔比平滑演算法

**需求 DASH-003**：計算全行四類產品貢獻佔比時，應使用以下平滑邏輯避免極端值：

1. 對各客戶 `productPreferences` 加權加總
2. 對原始比例取 0.9 次方（指數平滑）
3. 設定最小值 8%、最大值 55%
4. 重新正規化至總和 100%
5. 整數四捨五入，餘數依序補足至最大貢獻項目

### 4.5 客戶詳情檢視

#### 4.5.1 檢視模式

**需求 DETAIL-001**：客戶詳情支援兩種檢視模式，使用者可自由切換：

| 模式代號 | 模式名稱 | 說明 |
|----------|----------|------|
| `tabs` | 分頁檢視 | 資訊依功能分頁呈現，適合逐項深閱 |
| `onepage` | 單頁檢視 | 所有資訊垂直展開於單一頁面，適合快速總覽 |

#### 4.5.2 分頁檢視（tabs mode）分頁清單

**需求 DETAIL-002**：分頁檢視模式應包含以下九個分頁：

| 順序 | 分頁 ID | 分頁名稱 | 主要內容 |
|:----:|---------|----------|----------|
| 1 | `basic` | 基本資訊 | 姓名、英文名、職業資訊（產業、雇主、職稱、年資、雇用型態） |
| 2 | `contact` | 聯絡資訊 | 行動電話、市話、電子郵件、戶籍地址、通訊地址、居住地址、偏好聯絡方式 |
| 3 | `financial` | 財務 | 淨資產、負債、月收入、AUM 趨勢圖、資產配置環形圖 |
| 4 | `business` | 業務往來 | 持有產品樹（CustomerProductTree）、近一個月轉帳明細、近一個月信用卡授權明細、近三個月轉帳合計 |
| 5 | `preferences` | 偏好資訊 | 產品偏好百分比、消費類別偏好百分比、通路偏好、行銷同意狀態 |
| 6 | `tags` | 客戶標籤 | 結構化標籤（依分類群組顯示）與一般標籤 |
| 7 | `interactions` | 互動紀錄 | 客戶事件時間軸（依時間降冪），含狀態徽章 |
| 8 | `risk` | 風險 | 風險評分、等級、各類風險旗標（黑名單、法院扣押、催收、保證異常等） |
| 9 | `rating` | 評級 | VIP 等級、AUM 門檻說明、終身價值等級（lifetimeValueTier） |

#### 4.5.3 單頁檢視（onepage mode）區塊清單

**需求 DETAIL-003**：單頁檢視應以垂直卡片堆疊形式呈現，包含以下區塊（順序如下）：

1. 客戶摘要列（姓名、VIP 等級徽章、風險分數、帳戶狀態）
2. 行動建議方案（3 張卡片橫排，含意圖分數、業務話術、推薦產品）
3. 基本資訊 + 聯絡資訊（雙欄）
4. 財務概覽（資產配置圖 + AUM 趨勢圖）
5. 業務往來（三欄等高版面：左欄產品樹、中欄轉帳明細、右欄信用卡授權明細）
6. 偏好資訊
7. 客戶標籤
8. 風險資訊
9. 互動紀錄

#### 4.5.4 業務往來分頁 — 資料規格

**需求 DETAIL-004**：業務往來分頁應顯示以下三類資料：

| 資料集 | 筆數 | 時間範圍 | 顯示方式 |
|--------|:----:|:--------:|----------|
| 近一個月轉帳明細 | 20 筆 | 最近 30 天 | 可捲動列表 |
| 近一個月信用卡授權明細 | 25 筆 | 最近 30 天 | 可捲動列表 |
| 近三個月轉帳合計 | 60 筆計算 | 最近 90 天 | 顯示加總金額（NT$） |

**需求 DETAIL-005**：轉帳明細每筆需包含欄位：商家名稱（`merchant`）、金額（`amount`，NT$ 格式）、交易時間（`time`，yyyy/M/d HH:mm 格式）。

**需求 DETAIL-006**：信用卡授權明細每筆需包含欄位：商家名稱（`merchant`）、金額（`amount`）、交易時間（`time`）。

**需求 DETAIL-007**：交易明細依時間降冪排列（最新交易在最上方）。

#### 4.5.5 AUM 趨勢圖規格

**需求 DETAIL-008**：AUM 趨勢圖應為 SVG 長條圖，顯示最近 12 個月的月末 AUM，每月以存款（teal）與財管（cyan）堆疊長條呈現：

- X 軸：12 個月標籤（M1 至 M12 或月份名稱）
- Y 軸：AUM 金額（自動選擇萬 / 億作為單位）
- 入場動畫：700ms ease-out cubic 緩動，資料變更時重新播放
- 互動：滑鼠 hover 顯示當月存款、財管、合計 tooltip

#### 4.5.6 資產配置環形圖規格

**需求 DETAIL-009**：資產配置圖為互動式環形圖（DonutInteractive），各產品類別以不同色塊呈現：

| 類別 | 顏色 |
|------|------|
| 存款 | `#14b8a6`（teal-500） |
| 信用卡 | `#06b6d4`（cyan-500） |
| 基金 / 投資 | `#34d399`（emerald-400） |
| 貸款 | `#0ea5e9`（sky-500） |

- 環形圖中央顯示最高佔比類別名稱與百分比
- 點擊各區段 highlight 並顯示詳細數值

### 4.6 資料匯出功能

#### 4.6.1 功能說明

**需求 EXPORT-001**：系統提供客戶名單匯出功能，入口位於篩選名單模組。

#### 4.6.2 匯出設定

**需求 EXPORT-002**：匯出對話視窗提供下列設定選項：

| 設定項目 | 選項 |
|----------|------|
| 匯出範圍 | 篩選後結果 / 全部客戶 |
| 檔案格式 | CSV / JSON |
| 敏感資料遮罩 | 勾選後遮罩：電話、Email、帳號、卡號 |

**需求 EXPORT-003**：匯出欄位包含：`id`、`name`、`phone`、`email`、`accountNumber`、`creditCard`、`vipLevel`、`riskLevel`。

**需求 EXPORT-004**：CSV 格式需加 BOM（`\uFEFF`）以確保 Excel 正確顯示中文字元；欄位值含逗號時以雙引號包裹並跳脫。

**需求 EXPORT-005**：JSON 格式採縮排 2 格之 pretty-print 輸出。

**需求 EXPORT-006**：匯出檔案命名規則：`cus_export_YYYY-MM-DD.csv`（或 `.json`），日期為匯出當日。

**需求 EXPORT-007**：匯出動作透過瀏覽器 Blob + `URL.createObjectURL` 觸發下載，無需後端介接。

### 4.7 智能小助手（AI 對話）

#### 4.7.1 功能概述

**需求 AI-001**：系統提供浮動式 AI 對話視窗（右下角固定），讓行員在查看客戶詳情時即時取得 AI 建議。

#### 4.7.2 介面規格

**需求 AI-002**：AI 對話視窗規格如下：
- 預設收起，點擊 FAB 按鈕展開
- 展開後寬度 24rem（行動裝置最大 92vw）
- 對話紀錄維持於本次 session，關閉後清除

#### 4.7.3 快捷問句

**需求 AI-003**：系統提供三個預設快捷問句按鈕：

| 按鈕文字 | 功能說明 |
|----------|----------|
| 客戶摘要 | 生成選定客戶之 360 度摘要 |
| 問候話術 | 產生本次拜訪之開場問候建議詞 |
| 產品推薦 | 依客戶偏好與意圖推薦最適合產品 |

#### 4.7.4 LLM 整合

**需求 AI-004**：系統優先呼叫遠端 LLM API，失敗或未設定時自動降版為本機啟發式生成器（heuristic fallback），確保功能不中斷。

**需求 AI-005**：LLM 連線參數透過環境變數設定：

| 環境變數 | 說明 |
|----------|------|
| `VITE_LLM_ENDPOINT` | API Endpoint URL |
| `VITE_LLM_API_KEY` | API 金鑰（不可提交至版本控制） |
| `VITE_LLM_MODEL` | 模型識別碼（如 `gpt-4o-mini`） |

**需求 AI-006**：對話介面顯示思考中動畫（`思考中 • • •`），訊息回傳後消失。

**需求 AI-007**：助手訊息支援結構化格式解析，以粗體標題呈現各段落（偵測以「：」結尾或 emoji 為首的行）。

## 5. 資料需求

### 5.1 客戶主檔欄位定義

#### 5.1.1 身分識別欄位

| 欄位名稱 | 資料型別 | 格式規則 | 說明 |
|----------|----------|----------|------|
| `id` | String | `C` + 3 位數字（C001–C999） | 系統唯一客戶編號 |
| `name` | String | 中文全名 | 全資料集不得重複 |
| `nameEn` | String | 大寫威妥瑪拼音（SURNAME GIVENNAME） | 英文姓名 |

## 6. 商業規則

### 6.1 VIP 等級分類準則

**規則 BR-VIP-001**：VIP 等級依可投資資產（AUM）劃分如下（參照台灣主要銀行實務標準）：

| VIP 等級 | 顯示名稱 | AUM 門檻 | 顯示色系 |
|----------|----------|----------|----------|
| `VVVIP` | 超級貴賓 | AUM ≥ NT$3,000 萬 | Fuchsia |
| `VVIP` | 特級貴賓 | NT$1,000 萬 ≤ AUM < NT$3,000 萬 | Purple |
| `VIP` | 貴賓 | NT$300 萬 ≤ AUM < NT$1,000 萬 | Blue |
| `normal` | 一般客戶 | AUM < NT$300 萬 | Gray |

### 6.2 信用評分與風險等級對應

**規則 BR-RISK-001**：信用評分（riskScore）與風險等級（riskLevel）對應如下：

| riskScore | riskLevel | 說明 |
|-----------|-----------|------|
| `A+` | `low` | 信用優良，無違約紀錄 |
| `A` | `low` | 信用良好 |
| `B` | `medium` | 信用一般，需持續觀察 |
| `C` | `high` | 信用異常，有逾期或催收紀錄 |

**規則 BR-RISK-002**：VIP 等級與風險等級一致性要求：

| VIP 等級 | 允許 riskScore | 帳戶狀態 |
|----------|----------------|----------|
| VVVIP | 僅 `A+` | 必須 `active` |
| VVIP | `A+` 或 `A`，極少 `B` | 必須 `active` |
| VIP | `A+`、`A`、`B` | 通常 `active` |
| normal | 全部 | `active` 或 `失效戶` |

### 6.3 行動建議方案（推薦引擎）

**規則 BR-REC-001**：行動建議方案系統依客戶意圖標籤、產品偏好與消費習慣，產生以下三類洞察卡：

| `opening` | 推薦話術（開場白） |
| `recs` | 推薦產品列表（`[{name, reason}]`） |

**類型二：產品洞察（Product Insight）**

| 欄位 | 說明 |
|------|------|
| `title` | 產品類別 |
| `score` | 偏好分數（0–100） |

## 8. 安全與個資合規需求

### 8.1 個資遮罩規範

本節依台灣個人資料保護法（個資法）及 PCI DSS 標準制定。

**需求 SEC-MASK-001**：系統提供全域個資遮罩開關（`showMaskedData`），啟用時所有敏感欄位自動套用遮罩：

| 資料類型 | 欄位 | 遮罩規則 | 遮罩範例 |
|----------|------|----------|----------|
| 身分證字號 | `idCard` | 保留首 1 碼 + 後 4 碼，中間 5 碼以 `*` 取代 | `A*****6789` |
| 信用卡號 | `creditCard` | 僅顯示後 4 碼，前 12 碼以 `****-****-****-` 取代 | `****-****-****-3456` |
| 存款帳號 | `accountNumber` | 保留前 5 位數字 + 後 4 位數字，中間以 `*` 取代 | `00001*******9012` |
| 行動電話 | `phone` | 保留前 4 碼 + 後 3 碼 | `0912-***-456` |
| 電子郵件 | `email` | 保留首字母，`@` 前其餘以 `****` 取代 | `w****@example.com` |
| 地址（門牌號） | `address` | 數字類門牌以 `*` 取代 | `忠孝東路***號***樓` |

**需求 SEC-MASK-002**：遮罩邏輯應集中於 `maskValue(label, value, shouldMask)` 函式，依 label 關鍵字判斷遮罩類型，不得分散於各渲染元件內。

**需求 SEC-MASK-003**：`CustomerProductTree` 元件中帳號欄位（label 含「帳號」）的遮罩透過 `maskAcctNum()` 函式實作，保留前 5 位數字 + 後 4 位數字，中間以 `*` 取代。

**需求 SEC-MASK-004**：匯出功能啟用遮罩選項時，電話、Email、帳號、卡號欄位於匯出檔案中同樣套用遮罩。

### 8.2 角色存取安全

**需求 SEC-RBAC-001**：儀表板模組僅 `manager` 角色可存取，`specialist` 角色的 API 請求或直接 URL 存取均應被拒絕。

**需求 SEC-RBAC-002**：使用者登出後，`currentRole`、`selectedCustomer`、`assistantMessages` 等 session 狀態應完全清空。

### 8.3 API 金鑰安全

**需求 SEC-API-001**：LLM API 金鑰（`VITE_LLM_API_KEY`）透過環境變數注入，嚴禁 hard-code 於原始碼，且 `.env` 檔案不得提交至版本控制系統（需加入 `.gitignore`）。

### 8.4 資料安全

**需求 SEC-DATA-001**：系統為 Demo / Proof-of-Concept 用途，使用完全虛構的模擬資料，不連接任何真實客戶資料庫。

**需求 SEC-DATA-002**：若正式串接後端 API，所有傳輸應使用 HTTPS/TLS 1.2 以上協定。

## 11. 名詞定義

| 名詞 | 定義 |
|------|------|
| AUM | Assets Under Management，可管資產，即客戶在本行可投資資產總額 |

| NPL | Non-Performing Loans，不良貸款率 |
| LCG | Linear Congruential Generator，線性同餘亂數生成器，用於確定性 mock 資料生成 |
| PCI DSS | Payment Card Industry Data Security Standard，支付卡產業資料安全標準 |

| SRS | System Requirements Specification，系統需求規格書 |
| FAB | Floating Action Button，浮動操作按鈕（智能小助手觸發鈕） |
| CRM | Customer Relationship Management，客戶關係管理 |
| SPA | Single Page Application，單頁應用程式 |
